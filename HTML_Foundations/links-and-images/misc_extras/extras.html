MORE HTML ATTRIBUTES

Now that we're more comfortable with HTML attribute syntax, we can add some finishing touches to our HTML boilerplate code. Every web page you create should define the language it's written in and it's character set.

DOCUMENT LANGUAGE

A web page's default language is defined by the lang attribute on the top level html element. Out document is in English, so we'll use the en country code as the attribute value. If you aren't sure on your country code, look it up here country codes

CHARACTER SETS

A character set is kind of like a digital alphabet for your browser. It's different from the language your document in that it only affects how the letters themselves are rendered, not the language of the content.

  1. bir
  2. iki
  3. üç
  4. dört
  5. beş

When this is viewed in your browser, you can sometimes see weird stuff with the characters. This is because the default character set for most browsers don't accommodate these "special" characters. To fix this you specify UTF-8 character encoding by adding a meta element with a charset attribute to the head of the file

These days, UTF-8 is a sort of universal alphabet for the Internet. Every web page you create should have this line in its head.

HTML ENTITIES

This isn't related to images or links as such but needs discussed before moving onto CSS. An HTML entiry is a special character that can't be represented in plain text in a HTML document. This typically means it's a reserved character in HTML or you don't have a key on your keyboard for it.

RESERVED CHARACTERS

The "<" ">" and "&" are called reserved characters because they aren't allowed to be inserted into a html document without being encoded. This is because they mean something in the HTML syntax. 1 starts a tag, 2 ends a tag and 3 sets of a HTML entity.

There are three reserved characters in HTML: < > and &. You should always use HTML entities for these three characters.

Entities always begin with an ampersand and end with a semicolon, In between you put a special code that your browser will interpret as a symbol. In this case, it interprets lt, gt, and amp as less-than, greater-than, and ampersand symbols, respectively.

There's a crap-ton of HTML entities, explore here HTML entities

QUOTES

Curly quotes aren't necessary, however if you care about typography they'll be some of the most common HTML entities you'll use. There's four different kinds of curly quotes (opening and closing single and double quotes):

  • “
  • ”
  • ‘
  • ’

If you’re into “web typography,” you’ll also find yourself using curly quotes quite a bit.

Unlike straight quotes, these curly quote entities should hug the text.

COMMIT MESSAGES

Good commits are useful whether that be helping keep track of changes, making it easier for others to understand what changes were made, or even to help you remember your thought process when you've been away from a prpoject for a while.

A bad commit: fix a bug

An effective commit consists of 2 seperate parts: a subject, and a body:

Subject: A brief summary of the change you made to the project: This is the change I made to the codebase

Github has a 72-character limit, so keep commit subjects within this amount

Body: A concice yet clear description of what you did

  1. Describe the problem your commit solves and how.

Now that we learned the secret to creating a good commit message, let's try and fix the commit message from earlier:

  1. Add missing link and alt text to the company's logo
  3. Screen readers won't read the images to users with disabilities without this information

This is better as now developers can gain a better understanding of this commit message as it does the following:

  • Provides a subject that specifies the code's action
  • Contains a body that provides a concise yet clear description of why the commit needed to be made
  • Seperates the subject from the body with a new/blank line. This is a best practice. It makes commit messages easier for other developers to read

How to commit with a subject and body in the message

at first you will start off by using git commit -m message. To make a commit with subject and body in message, the simplest way is to type git commit without the -m flag and message argument

doing so will open a new VSCode tab if you have set VSCode as the Git editor

When to commit

It's good to view a commit like a snapshot of your code at the moment it was made That version of your code up to that point will be saved for you to revent back to or look back at

when writing code, it's best practice to commit every time you make a meaningful change in the code. This will create a timeline of your progress and show that your finished code didn't appear out of nowhere

In other words, make a commit if you get a piece of code you are working on to function like you want it to, fix a typo, or fix a bug.

There will come a time when you are working on a project and you FINALLY get something just right, this would be a good time to commit, and then 30 seconds to a few days later it breaks. You have no idea what you changed, everything looks to be the same and you don't recall editing that line. You'd be capable of going back through your commit history and either revert your code back to the last commit you made when you first got that part working or go back and see what your code looked like at that point in time
